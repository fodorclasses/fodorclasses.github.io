Git Lab: State, Flow, and Conflict

A hands-on exercise designed to make Git’s conceptual model stick: state (where files are) + flow (which commands move them). You will intentionally create a merge conflict and resolve it.

Time: 25–40 minutes Format: pairs Tools: Git, GitHub account, terminal Outcome: one resolved conflict

Learning goals

  • Identify where your changes are: working directory, staging area, local repo, remote repo.
  • Use git status to diagnose what to do next.
  • Experience a merge conflict and resolve it calmly.
Mindset: a merge conflict is not an error. It’s Git refusing to guess what you mean.

Roles

Driver types commands. Observer narrates what state you’re in and what each command does.

Switch roles after Part 2 (after both commits are made).

The state + flow model (keep this in your head)

Working Directory → Staging → Local Repo → Remote Repo
     git add         git commit      git push
If you get lost, run git status and ask: “Which arrow am I on?”

Quick checklist

Checkboxes are for you. They won’t be saved.

Part 0: Setup (5 minutes)

  1. Driver creates a new GitHub repo named git-lab-yourname (public or private is fine).
  2. Driver adds Observer as a collaborator.
  3. Both students clone the repo locally:
git clone https://github.com/<username>/git-lab-yourname.git
cd git-lab-yourname
Checkpoint question: Is your working directory currently empty but linked to a remote?

Part 1: Normal workflow (confidence builder)

Driver creates notes.txt with exactly one line:

This file belongs to science.

Then run:

git status
git add notes.txt
git commit -m "Add notes file"
git push

Observer pulls the changes:

git pull
Checkpoint questions:
  • Where was the file before git add?
  • Where did it move after git commit?
  • Where is it after git push?

Part 2: Divergence (the trap is set)

Now both partners will edit the same line in notes.txt without pulling first.

Important: Do not run git pull before committing your edit. This lab only works if histories diverge.

Driver change

Change the line to:

This file belongs to computational science.

Observer change

Change the same line to:

This file belongs to experimental science.

Now both partners run (each on their own machine):

git add notes.txt
git commit -m "Update notes"
Checkpoint question: How many different versions of history exist right now?

Switch roles now: whoever was Driver becomes Observer and vice versa.

Part 3: The collision (guaranteed failure)

  1. New Driver pushes first:
git push
  1. New Observer tries to push:
git push
Expected result: push is rejected because the remote has commits you do not have locally. Git is protecting history from being overwritten.
Instructor prompt: Is this an error, or Git being cautious on your behalf?

Part 4: Fetch, merge, conflict

On the machine that failed to push, run:

git fetch
git status
git merge origin/main

You should now see a merge conflict. Open notes.txt and find the conflict markers:

<<<<<<< HEAD
This file belongs to experimental science.
=======
This file belongs to computational science.
>>>>>>> origin/main
Checkpoint questions:
  • Which version is HEAD?
  • Which version came from origin/main?
  • Why can’t Git decide automatically?

Part 5: Resolve and synchronize (the click)

Edit notes.txt to a single line that resolves the disagreement, for example:

This file belongs to both experimental and computational science.

Then finish the merge and push:

git add notes.txt
git commit -m "Resolve merge conflict"
git push

On the other machine, pull the resolution:

git pull
Success condition: both machines show the same line in notes.txt, and git status says “nothing to commit”.

Part 6: Reflection (2 minutes)

Answer briefly (complete sentences). If you print this page, write on paper.

1) What state were you in when Git stopped you from pushing?

2) Why is git status the most important command?

3) What would happen if Git allowed you to push anyway?

Tip: You can also save these answers elsewhere. This page does not store them.

Troubleshooting (minimal, but useful)

  • If git merge origin/main says it can’t find origin/main, run git branch -a and check the remote branch name (sometimes it’s origin/master).
  • If authentication fails, GitHub may require a token instead of a password. Use the browser prompt Git opens, or follow your course’s token instructions.
  • If you truly get stuck: run git status, read it out loud, and do what it tells you.
The lab is successful when you see a conflict once and survive it. That’s the whole spell.
